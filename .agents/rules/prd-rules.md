# AI-Native OS: Comprehensive Architecture & Implementation Rules

## Core Philosophy:
This system is an AI-Native Enterprise Resource Planning (ERP) platform. It MUST replicate the deep, robust "Meta-Engine" architecture of the Frappe Framework (metadata-driven schemas, generic workflows, relational child tables) while leveraging the asynchronous speed, scale, and AI-native capabilities of a Python FastAPI Modular Monolith and PostgreSQL (Supabase).

We are NOT hand-coding individual UI pages for hundreds of ERP features. We are building a dynamic engine that renders them based on backend schemas.

---

## 1. The Meta-Engine Architecture (The Foundation)

Instead of hardcoding tables, the system must be built on a generic Metadata architecture:

* **DocType Schemas (Pydantic/FastAPI):** Every entity must be defined by a strict Pydantic schema and SQLAlchemy model, engineered by reading the original Frappe workspace requirements.
* **Modular Monolith Backend:** The backend MUST be built strictly using Python and FastAPI. All modules (HR, Finance, Sales) reside in a single codebase but are strictly isolated via FastAPI APIRouter. No direct cross-module database joins are allowed; cross-domain communication happens via internal API routing.
* **Dynamic UI Rendering:** Build generic frontend components (using React/Vue, TanStack, etc.) that automatically consume the OpenAPI/JSON schema generated by FastAPI to render high-density lists and complex forms dynamically.



---

## 2. Microscopic Schema Replication (Business Data)

When replicating a Frappe module, the AI MUST physically map the architecture microscopically:

* **Foreign Keys:** Frappe Link fields must become strict UUID REFERENCES in Postgres.
* **Child Tables:** Frappe Table fields (nested grids) MUST be implemented as independent relational Postgres tables with ON DELETE CASCADE foreign keys back to the parent. Never flatten hierarchical data into JSONB arrays.
* **AI Vector Data:** Utilize the pgvector extension alongside standard relational columns to store high-dimensional embeddings natively within the same PostgreSQL tables.

---

## 3. Standardized Database Integrity (Supabase/PostgreSQL)

* **Single Tenant Table Structure:** All tenants share a single database, isolated universally by Postgres Row-Level Security (RLS) linked to custom JWT claims (tenant_id). Never use separate databases per tenant.
* **The docstatus State Machine:** EVERY transactional table MUST include the Frappe-standard docstatus integer: 0 (Draft), 1 (Submitted), 2 (Cancelled), serving as the backbone for the system's human-in-the-loop approval workflows.
* **Audit Logging:** A universal audit_logs table must track every Create/Update/Delete action across the system (User ID, Entity Type, JSON diff).

---

## 4. Core "Hub" Master Data
All modules (HR, Finance, Sales) must link to a central nervous system. Do not duplicate master data globally.
* **System Masters:** Global definitions for Currencies, Countries, and Holiday Lists.
* **Org Masters:** Unified departments, designations, cost_centers, and branches.

---

## 5. The AI Orchestration Engine (The "OS Brain")
The AI is not a chatbot; it is a transactional agent executing backend tools.
* **Python-First AI:** MUST use Python-native frameworks (LangGraph and CrewAI) to ensure deep, native integration with the FastAPI backend, SQLAlchemy ORM, and data processing libraries.
* **The Supervisor-Worker Pattern:**
    * **Supervisor (LangGraph):** Manages the stateful overarching workflows, deterministic routing, and human-in-the-loop interruptions.
    * **Domain Agents (CrewAI):** Specialized autonomous workers (e.g., Financial Analyst, HR Manager) armed strictly with the exact same Pydantic schemas and FastAPI endpoints used by the Meta-Engine.
* **Deterministic Validation:** The AI can only insert or update data if its JSON output passes the identical Pydantic validation schema that a human user would face.



---

## 6. Agile Development Phases (Engine First)
* **The Core Engine & Master Data:** Build the FastAPI foundation, the docstatus workflow logic, and the foundational Supabase Master Data tables with RLS.
* **Vertical Slice MVP (Employee + Leave):** Microscopically replicate the Core Employee Lifecycle and Leave Management modules to prove the FastAPI Meta-Engine handles complex child tables and routing perfectly.
* **AI Supervisor Assembly:** Connect the LangGraph Orchestrator to the MVP modules. Prove the AI can autonomously draft a Leave Application via the API and trigger the docstatus state machine (Draft mode) for human review.
* **Horizontal Scaling:** Systematically map and deploy the remaining modules (Payroll, Finance, Appraisals) using the proven Python Modular Monolith pattern.
